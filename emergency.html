Step 6 of 8: Emergency Workflow

Overview: What Happens After "Fall Detected"

When our fall detection algorithm confirms a fall candidate, a carefully orchestrated emergency workflow begins. This end-to-end process prioritizes user safety while preventing false alarms through human-in-the-loop confirmation:

  1. Detector raises a fall candidate based on the 3-stage rule logic
  2. Show "I'm OK?" dialog with countdown (10–15 s) and text-to-speech announcement
  3. If not canceled → acquire GPS location and compose SMS with coordinates [6]
  4. Send emergency message to up to 3 contacts with escalation policy
  5. Optional escalation if no acknowledgment received within 60–90 s
  6. Log the event and rate-limit further alerts for ≥ 5 min

Design Principles

  • Privacy-first: No video or images transmitted—only text and GPS coordinates
  • User confirmation: Always allow cancellation to prevent false alarms
  • Reliability: Multiple delivery paths with retry logic and escalation
  • Accessibility: Voice prompts, haptic feedback, and large touch targets

Countdown Interface

⏰ "Are you OK?"

Apple-style confirmation dialog with 10–15 s countdown, TTS announcement, and haptic feedback.

  • Large modal with animated countdown ring
  • Voice prompt: "A fall was detected. Tap cancel if you're okay."
  • Haptic buzz at start and 5s warning
  • WCAG AA accessibility compliance

SMS Delivery [6]

📱 Emergency Alert

Multiple delivery paths with GPS coordinates and escalation to 3 contacts.

  • SmsManager: Direct SMS (fastest)
  • Intent: System SMS app (Play-friendly)
  • Server relay: Twilio/AWS backup
  • Rate limiting: ≥ 5 min between alerts

Privacy & Safety

🔒 Private

Privacy-first design with no video transmission and local-only processing.

  • No images/video leave device
  • Only GPS + text during emergencies
  • Local encryption with Android Keystore
  • Large cancel button prevents false alarms
